BrokeBot: Unified Full Development Plan

Prepared on April 9, 2026

1. Product Goal

BrokeBot is a personal local-first job search and application assistant built to help you discover, review, tailor, and submit applications quickly without burning money on infrastructure. It should automate the repetitive parts of job hunting while keeping the final high-risk step under your control whenever a platform becomes unpredictable.

2. Final Product Decisions

Core decisions after merging both plans:
  • Use Node.js for backend services and automation orchestration.
  • Use React + Tailwind CSS for a simple local dashboard.
  • Use Playwright for scraping, PDF rendering, and form automation.
  • Use OpenAI gpt-4o-mini as the only AI model.
  • Use SQLite with better-sqlite3 for all persistence.
  • Use a resume JSON source of truth to reduce token spend.
  • Generate a PDF resume as the primary application artifact using HTML + Playwright.
  • Keep DOCX export as a later compatibility enhancement, not a day-one requirement.
  • Run the app locally for a single user only; no auth, no cloud infra, no multi-tenant concerns.

3. Why These Decisions Win

  • PDF is the safest default for uploads across LinkedIn, Indeed, Greenhouse, Lever, and most external forms.
  • HTML-to-PDF with Playwright keeps formatting fully under our control and costs nothing per document.
  • Using resume JSON avoids repeated parsing and sharply reduces prompt size.
  • gpt-4o-mini gives better extraction and writing quality than the older 3.5 family while still being very cheap.
  • SQLite is enough for a personal automation system and keeps the app portable and inspectable.
  • Pause-before-submit mode protects you from dangerous blind automation on multi-step or unstable external forms.

4. Unified Architecture

Frontend: React dashboard for settings, queue management, previews, and run status.

Backend API: Express server exposing settings, jobs, application queue, AI generation, and bot execution endpoints.

Automation Layer: Playwright workers for scraping, PDF generation, and apply flows.

AI Layer: Prompt builder + OpenAI client + validation/parsing + token accounting.

Database Layer: SQLite repository functions for jobs, applications, settings, logs, and counters.

Layer Responsibility Key Output
React UI Settings, queue review, previews, approvals, metrics Manual control surface

5. Final Feature Scope

5.1 Resume and Application Package Engine

  • User uploads a base resume once.
  • Backend converts the resume into structured JSON and stores it in settings.
  • For each job, the AI generates:
    • Three tailored bullet rewrites or improvements.
    • A short cover letter with a hard cap.
    • Detected hidden keywords or “humanity check” strings from the JD.
  • The tailored resume is rendered into HTML and exported to PDF with Playwright.
  • The user can preview resume PDF content and cover letter in the queue before applying.

5.2 Job Discovery and Scraping

  • Primary sources: LinkedIn, Indeed, Remote.com.
  • Secondary review-only targets: Amazon, Greenhouse, Lever, other external application pages.
  • Search terms include remote software, full stack, frontend, and technical support variants.
  • Scrapers store title, company, location, posted date, salary text, JD content, apply link, and platform flags.
  • Jobs older than 7 days are skipped immediately to save time and tokens.
  • Remote roles are preferred; onsite roles must pass salary threshold logic or be sent to manual review.

5.3 Review and Apply Flow

  • Scraped jobs land in a queue with statuses like new, ready, review_required, skipped, and applied.
  • High-confidence direct-apply flows can be auto-filled.
  • External forms enter pause-before-submit mode so you confirm the final click.
  • A daily limit is enforced using a resettable counter in SQLite.
  • If the result is ambiguous, BrokeBot marks the attempt as uncertain for manual review.

5.4 Alerts and Monitoring

  • Dashboard shows jobs found today, jobs ready to apply, applied today, failures, and estimated token spend.
  • Optional CallMeBot WhatsApp notification for high-value or blocked jobs.
  • Console and UI alerts for CAPTCHA, expired session, missing fields, and failed submissions.

6. Prompt Strategy and Token Control

  • Store the base resume as compact JSON once and never re-parse unless the resume changes.
  • Clean the JD before sending it to AI:
    • Strip boilerplate HTML.
    • Remove generic benefits and culture sections where possible.
    • Truncate to a safe size if still too long.
  • Use one structured AI request per job to return:
    • Tailored bullet updates
    • Short cover letter
    • Hidden keyword / secret word findings
    • Optional answer hints for common application questions
  • Require valid JSON response shape and validate it before accepting output.
  • Track prompt and completion tokens per request and aggregate spend in SQLite.

7. Business Rules

7.1 Date Filter

  • If the job was posted more than 7 days ago, skip it immediately.
  • If the platform filter supports “Past 24 hours” or “Past week,” use it before scraping full details.

7.2 Salary and Location Filter

  • Remote jobs are allowed by default, subject to role filtering.
  • Onsite jobs are only eligible when salary data is present and passes the configured threshold.
  • If salary is missing for onsite jobs, mark them as review required.
  • If salary currency is unclear, store the raw text and require review.
  • If a range is given, use midpoint logic for threshold comparison.

7.3 Sponsorship Logic

  • If location is India: authorized to work = yes; sponsorship required = no.
  • If location is outside India: authorized to work = no; sponsorship required = yes.
  • These defaults must remain editable in settings because some applications phrase the question differently.

7.4 Role Filtering

  • Default include terms: software engineer, frontend engineer, full stack engineer, technical support, support engineer.
  • Default skip terms: intern, unpaid, volunteer.
  • Default configurable skip terms: senior, staff, principal, lead.

7.5 Apply Safety Rules

  • Hard stop at 50 applications per day by default.
  • Handle at most 5 pages in a multi-step flow; bail on page 6 or beyond.
  • If submission confirmation is not detected, mark the result as uncertain.
  • If CAPTCHA or bot-blocking occurs, pause and notify you rather than forcing retries forever.

9. API Surface

  • GET /api/health for readiness checks.
  • GET /api/dashboard for summary counts and spend.
  • GET /api/jobs with filters for queue views.
  • POST /api/jobs/:id/approve to mark a job ready for application.
  • POST /api/jobs/:id/skip to reject a job.
  • POST /api/ai/generate/:jobId to generate the tailored package.
  • POST /api/apply/:jobId to launch an apply run.
  • POST /api/scrape/run to manually trigger scraping.
  • GET /api/settings and POST /api/settings for user configuration.
  • POST /api/resume/upload for base resume intake.

10. Frontend Scope

10.1 Dashboard

  • Jobs found today
  • Jobs ready to apply
  • Applied today
  • Daily limit progress
  • Estimated AI spend and budget warning when nearing the chosen threshold
  • Bot status cards for scraper and apply engine

10.2 Queue

  • Job list with filters by platform, status, remote, salary confidence, and age
  • Job detail pane with JD preview and extracted metadata
  • Tailored resume preview link and cover letter preview
  • Approve, skip, regenerate, and apply actions

10.3 Settings

  • OpenAI API key
  • Resume upload and resume JSON preview
  • Role keywords and exclusions
  • Salary threshold
  • Daily application limit
  • Phone number, work authorization, sponsorship defaults
  • LinkedIn and Indeed session status
  • Optional WhatsApp notification settings

11. Automation Flows

11.1 Scraper Run

  1. Open saved browser profile or session.
  2. Run search queries with date filters first.
  3. Collect visible job metadata.
  4. Discard jobs older than 7 days.
  5. Open candidate jobs and extract JD + salary + apply link.
  6. Deduplicate against existing records using URL and content hash strategies.
  7. Store jobs with initial review state.

11.2 AI Generation Run

  1. Fetch base resume JSON and cleaned JD.
  2. Send one structured prompt to gpt-4o-mini.
  3. Validate the JSON response.
  4. Generate HTML resume using tailored bullet content.
  5. Render PDF via Playwright.
  6. Store outputs and usage metrics.

11.3 Apply Run

  1. Verify daily limit has not been reached.
  2. Open the platform-specific flow.
  3. Upload tailored PDF if supported.
  4. Fill structured answers from settings and AI output.
  5. Handle short cover letter field limits by truncation.
  6. Submit automatically only on high-confidence flows.
  7. Pause before the final click for risky external forms.
  8. Detect confirmation or mark uncertain.

12. Edge Case Hardening

  • AI timeout: retry once, then fall back to base resume package and log the failure.
  • Invalid JSON: attempt extraction, then reject the generation if validation still fails.
  • Very long JD: aggressively clean and truncate before calling AI.
  • Expired platform session: trigger re-login guidance and stop the run.
  • CAPTCHA: pause, notify, wait for manual solve, then resume.
  • Already applied job: dedupe by URL plus fuzzy company-title-content hash.
  • Inaccessible JD: mark skipped with reason inaccessible.
  • Missing upload field: continue with text fields if the platform allows it.
  • Rate limit or 429: exponential backoff with capped retries.
  • “Secret word” or hidden prompt test: extract and store it in the application package for later answer filling.

13. Recommended Build Phases

Phase 1: Foundation

  • Create project structure, config management, SQLite schema, migrations, and Express API skeleton.
  • Add settings CRUD and health endpoints.
  • Deliverable: local app boots, DB initializes, settings are editable.

Phase 2: Resume Data and PDF Engine

  • Implement base resume ingestion and JSON storage.
  • Build HTML resume template and Playwright PDF generator.
  • Deliverable: generate a polished PDF from stored resume JSON.

Phase 3: AI Package Generator

  • Build JD cleaner, prompt builder, OpenAI client, validation, and cost tracking.
  • Deliverable: per-job tailored bullets, short cover letter, hidden keyword extraction, and generated PDF.

Phase 4: React Review UI

  • Create Settings, Queue, and Dashboard pages.
  • Deliverable: local control panel for review and operations.

Phase 5: LinkedIn Scraper

  • Add saved login session, search automation, date filter, extraction, and dedupe.
  • Deliverable: LinkedIn jobs flowing into SQLite queue.

Phase 6: Indeed and Remote.com Scrapers

  • Implement second and third source adapters using the same normalized storage model.
  • Deliverable: multi-source job ingestion.

Phase 7: Apply Bot

  • Build LinkedIn Easy Apply, Indeed apply support, and external pause-before-submit flows.
  • Deliverable: controlled semi-automation for applications.

Phase 8: Scheduler, Alerts, and Polish

  • Add cron jobs, optional WhatsApp alerts, budget warnings, logs, and retry controls.
  • Deliverable: usable daily driver version.

14. Testing Strategy

  • Unit tests for salary parsing, date parsing, JD cleaning, JSON validation, and business rules.
  • Integration tests for database repositories and core API endpoints.
  • Playwright smoke flows against local fixtures before live-site automation.
  • Manual validation runs for LinkedIn and Indeed because live selectors drift often.

15. What Not to Build Yet

  • Multi-user auth
  • Cloud sync
  • Public SaaS billing
  • Too many job boards beyond the highest ROI sources
  • Fully blind submission on complex external flows
  • Fancy analytics beyond budget, volume, and queue usefulness

16. First Implementation Slice

The smartest place to begin is the foundation slice: project scaffold, SQLite schema, settings storage, Express API skeleton, and a health-checked local app shell. Once that exists, every later feature has a stable place to land.

17. Success Criteria for Version 1

  • You can store your resume data, preferences, and API key locally.
  • You can scrape recent jobs from at least one platform into a review queue.
  • You can generate a tailored PDF and short cover letter for a selected job.
  • You can approve a job and launch an apply flow.
  • The bot stops safely on risky flows instead of guessing.
  • You can see daily counts, spend, failures, and uncertain submissions in the dashboard.

18. Recommended Immediate Next Step

Start with Phase 1 only: create the local monorepo structure, initialize SQLite, define the schema, stand up the Express server, and build the settings endpoints. This gives us the foundation to attach the PDF engine, AI package generator, and scrapers without rework.
